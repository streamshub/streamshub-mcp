# 5.1 E2E Tests for New CRD Tools

## Context

All new CRD tools from Phase 2 need end-to-end verification on a real Kind cluster with actual Strimzi deployments. The existing `systemtest` module provides the framework — Kind cluster setup, Strimzi deployment, Prometheus integration, Allure reporting.

## Test Infrastructure

### Additional CRD Deployments Needed

Add to Kind cluster setup (`dev/scripts/setup-strimzi.sh` or systemtest setup):

1. **KafkaConnect** — deploy Connect cluster with `build` configuration (for connector image building) or pre-built image with a test connector (e.g., FileStreamSource)
2. **KafkaConnector** — deploy at least one connector (FileStreamSource or MirrorMaker connector)
3. **KafkaUser** — deploy users with different auth types (SCRAM-SHA-512, TLS) and various ACL rules
4. **KafkaRebalance** — deploy with Cruise Control enabled on Kafka cluster, create a rebalance proposal
5. **KafkaMirrorMaker2** — requires a second Kafka cluster; deploy MM2 with source→target mirror
6. **KafkaBridge** — deploy Bridge pointing to test cluster
7. **DrainCleaner** — deploy Strimzi Drain Cleaner with webhook

### Kustomize Manifests for Test Resources

Create `systemtest/src/test/resources/strimzi/` with CR manifests for each resource type.

## Test Classes

### `KafkaConnectToolsE2ETest.java`
- Deploy KafkaConnect CR → wait for Ready
- `list_kafka_connects` → verify Connect cluster appears with correct status
- `get_kafka_connect` → verify config, replicas, image, bootstrap servers
- `get_kafka_connect_pods` → verify pod count matches replicas
- `get_kafka_connect_logs` → verify log retrieval works

### `KafkaConnectorToolsE2ETest.java`
- Deploy KafkaConnector CR → wait for Ready
- `list_kafka_connectors` → verify connector appears
- `list_kafka_connectors` with `connectCluster` filter → verify filtering works
- `get_kafka_connector` → verify class, tasksMax, config, status
- Test with paused connector → verify state reflected

### `KafkaUserToolsE2ETest.java`
- Deploy KafkaUsers with SCRAM and TLS auth → wait for Ready
- `list_kafka_users` → verify users appear with auth types
- `get_kafka_user` → verify ACL rules and quotas displayed
- Verify secret data is NOT exposed (only secret name)

### `KafkaRebalanceToolsE2ETest.java`
- Enable Cruise Control on Kafka cluster
- Create KafkaRebalance CR → wait for ProposalReady
- `list_kafka_rebalances` → verify state
- `get_kafka_rebalance` → verify optimization result, goals

### `KafkaMirrorMaker2ToolsE2ETest.java`
- Deploy second Kafka cluster + MM2 → wait for Ready
- `list_kafka_mirror_makers` → verify MM2 appears
- `get_kafka_mirror_maker` → verify source/target clusters, mirror config
- `get_kafka_mirror_maker_logs` → verify log collection

### `KafkaBridgeToolsE2ETest.java`
- Deploy KafkaBridge → wait for Ready
- `list_kafka_bridges` → verify bridge appears
- `get_kafka_bridge` → verify HTTP config, bootstrap servers

### `DrainCleanerToolsE2ETest.java`
- Deploy Strimzi Drain Cleaner
- `list_drain_cleaners` → verify deployment found
- `get_drain_cleaner` → verify webhook config
- `check_drain_cleaner_readiness` → verify production readiness assessment

### `ConfigurationToolsE2ETest.java`
- Deploy two Kafka clusters with different configs
- `get_kafka_cluster_config` → verify effective config includes ConfigMap content
- `compare_kafka_clusters` → verify diff highlights differences

## CI Integration

Update `.github/workflows/e2e.yml`:
- Add additional CRD deployment steps
- May need more cluster resources (second Kafka cluster for MM2)
- Consider parallel test execution for faster CI

1. All E2E tests pass on Kind cluster
2. Allure report shows all scenarios covered
