# 5.2 E2E Diagnostic Workflows with Injected Faults

## Context

Diagnostic tools are the most complex features — they orchestrate multi-step workflows with sampling and elicitation. E2E tests should verify they work against real clusters, including clusters with injected faults.

## Fault Injection Scenarios

### Scenario 1: Pod Crash (OOMKilled)
- Deploy Kafka cluster with deliberately low memory limits on one node pool
- Produce load to trigger OOM
- Run `diagnose_kafka_cluster` → verify it detects OOMKilled pod, correlates with restart count
- Verify diagnostic report includes the right root cause area

### Scenario 2: Unhealthy Connector
- Deploy KafkaConnect + KafkaConnector with invalid config (e.g., wrong topic name or unreachable source)
- Wait for connector to enter FAILED state
- Run `diagnose_kafka_connector` → verify it detects failed tasks, gets relevant logs
- Verify report identifies connector config as root cause

### Scenario 3: Under-Replicated Partitions
- Scale down a broker node pool (reduce replicas)
- Run `diagnose_kafka_metrics` → verify it detects under-replicated partitions
- Verify metrics correctly show non-zero under-replicated partition count

### Scenario 4: Operator Reconciliation Failure
- Deploy an invalid Kafka CR (e.g., reference non-existent ConfigMap)
- Run `diagnose_kafka_cluster` → verify it detects reconciliation error in operator logs
- Verify events show the failure

## Test Classes

### `DiagnosticWorkflowE2ETest.java`
- One test per fault scenario
- Setup: deploy fault condition
- Execute: call diagnostic tool
- Assert: report contains expected findings
- Teardown: restore healthy state

### Test without Sampling
- Verify diagnostic tools work when Sampling is not supported by client
- Should gather all data without LLM-guided triage
- Report should contain raw data in all sections

## Key Considerations

- Fault injection may take time — use appropriate wait conditions
- Some faults (OOM) are non-deterministic — use high memory pressure with low limits for reliability
- Clean up after each test to avoid affecting subsequent tests
- May need separate test profiles for fault injection (heavier resource requirements)

## Verification

1. All fault scenarios produce correct diagnostic reports
2. Tests are reliable (not flaky due to timing)
3. Cleanup is thorough — no fault state leaks between tests
