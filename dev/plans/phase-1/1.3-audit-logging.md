# 1.3 Audit Logging

## Context

No audit trail exists for tool invocations. For production, operators need to know who called what tool, when, with which arguments, and whether it succeeded. The auth architecture doc specifies structured JSON audit events.

**Depends on:** 1.1 (OIDC authentication / `McpSecurityContext`)

## What Changes

### New Files

**`common/src/main/java/io/streamshub/mcp/common/security/AuditFilter.java`**
- Implements `GuardrailFilter` with `@Priority(10)` (runs after rate limiting at 50, before input validation at 100)
- `filterInput()`: captures start time, records tool name and sanitized args
- `filterOutput()`: logs structured audit event with duration and result status
- Injects `McpSecurityContext` to get user identity
- Uses JBoss Logger (consistent with codebase) with a dedicated `audit` logger category

**Audit event structure:**
```json
{
  "timestamp": "2026-04-17T10:23:45Z",
  "userId": "honza@firma.cz",
  "email": "honza@firma.cz",
  "groups": ["kafka-dev", "ns-honza-rw"],
  "tool": "list_kafka_clusters",
  "arguments": {"namespace": "kafka-prod"},
  "result": "success",
  "durationMs": 234
}
```

- Arguments are sanitized: only parameter names and non-sensitive values logged (no secrets, tokens)
- On error: `result` = `"error"`, add `errorMessage` field
- For anonymous users (dev mode): `userId` = `"anonymous"`

### Modified Files

**`strimzi-mcp/src/main/resources/application.properties`** — add:
```properties
# Audit logging
mcp.audit.enabled=true
quarkus.log.category."io.streamshub.mcp.audit".level=INFO
```

### Test Files

**`common/src/test/java/io/streamshub/mcp/common/security/AuditFilterTest.java`**
- Test: successful tool call produces audit log entry with user, tool, args, duration
- Test: failed tool call produces audit log with error info
- Test: anonymous user logged as "anonymous"
- Test: sensitive arguments are not logged verbatim
- Test: audit disabled via config produces no log entries

## Key Patterns

- Follows existing `GuardrailFilter` pattern exactly — same interface, same CDI registration
- Filter priority 10 ensures audit happens even if later filters reject the request (rate limit at 50 runs first, so rate-limited calls are NOT audited — they never reach this filter). If we want to audit rate-limited rejections too, move audit to priority 1.
- Uses `LOG.infof()` pattern consistent with codebase (not `System.out` or `println`)
- Dedicated logger category allows routing audit logs to a separate sink in production
