# 1.2 Kubernetes User Impersonation

## Context

Currently the MCP server uses its ServiceAccount's own permissions for all K8s API calls. With OIDC auth (1.1) we know who the user is. This step propagates that identity to the K8s API via impersonation headers (`Impersonate-User`, `Impersonate-Group`), so K8s RBAC enforces the real user's permissions ŌĆö not the MCP server's.

**Depends on:** 1.1 (OIDC authentication / `McpSecurityContext`)

## What Changes

### New Files

**`common/src/main/java/io/streamshub/mcp/common/security/ImpersonatingKubernetesClientProducer.java`**
- `@ApplicationScoped` CDI producer
- Creates a wrapper around the default `KubernetesClient` that adds impersonation headers per-request
- When `McpSecurityContext` has a real identity: sets `Impersonate-User` to `userId`, `Impersonate-Group` for each group
- When anonymous (dev mode): passes through without impersonation ŌĆö uses SA directly
- Uses Fabric8's `RequestConfig` with `.withImpersonateUsername()` / `.withImpersonateGroups()`

**`common/src/main/java/io/streamshub/mcp/common/security/ImpersonationConfig.java`**
- `@ConfigMapping(prefix = "mcp.security.impersonation")`
- `enabled()` ŌĆö boolean, default `false`, set to `true` in prod
- `userClaim()` ŌĆö which JWT claim to use for user (default: `sub`)
- `groupsClaim()` ŌĆö which JWT claim to use for groups (default: `groups`)

### Modified Files

**`common/src/main/java/io/streamshub/mcp/common/service/KubernetesResourceService.java`**
- Currently injects `KubernetesClient` directly
- Change to inject the impersonation-aware client (producer qualifies it)
- Zero changes to method signatures ŌĆö impersonation is transparent

**`strimzi-mcp/install/003-ClusterRole.yaml`** ŌĆö when impersonation is enabled, the MCP SA needs:
```yaml
- apiGroups: [""]
  resources: ["users", "groups"]
  verbs: ["impersonate"]
```
Add this as an optional block, documented in README. Alternatively, create a separate `008-ClusterRole-impersonation.yaml`.

**`strimzi-mcp/src/main/resources/application.properties`** ŌĆö add:
```properties
# K8s impersonation (requires OIDC enabled)
mcp.security.impersonation.enabled=false
%prod.mcp.security.impersonation.enabled=${quarkus.oidc.enabled}
mcp.security.impersonation.user-claim=sub
mcp.security.impersonation.groups-claim=groups
```

### Test Files

**`common/src/test/java/io/streamshub/mcp/common/security/ImpersonatingKubernetesClientProducerTest.java`**
- Test: when identity present, client includes impersonation headers
- Test: when anonymous, client passes through without impersonation
- Test: when impersonation disabled, client passes through regardless of identity
- Mock `KubernetesClient` and verify `RequestConfig` settings

## Key Design Decisions

- **Fabric8 RequestConfig approach**: Fabric8 client supports `.withRequestConfig(rc -> rc.withImpersonateUsername(...))` on individual operations. However, this would require changing every call site. Instead, we produce a pre-configured client via CDI producer.
- **CDI alternative**: Use `@Produces @RequestScoped` to create a per-request client with impersonation baked in. This is cleaner but has performance implications (client creation per request). Evaluate with profiling.
- **Fallback**: If impersonation fails (403 from K8s ŌĆö SA lacks impersonate permissions), the error propagates as a `ToolCallException` with a clear message about missing RBAC.
