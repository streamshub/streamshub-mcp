# 3.3 OpenTelemetry Tracing

## Context

Diagnostic tools make 5-10+ K8s API calls per invocation. Without tracing, debugging slow responses requires manual log correlation. OpenTelemetry provides distributed tracing with spans for each K8s call.

## What Changes

### Dependencies

**`strimzi-mcp/pom.xml`** — add:
```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-opentelemetry</artifactId>
</dependency>
```

### Configuration

**`strimzi-mcp/src/main/resources/application.properties`:**
```properties
# OpenTelemetry (disabled by default)
quarkus.otel.enabled=false
%prod.quarkus.otel.enabled=true
quarkus.otel.exporter.otlp.endpoint=http://localhost:4317
quarkus.otel.service.name=strimzi-mcp

# Trace propagation
quarkus.otel.propagators=tracecontext,baggage
```

### Auto-Instrumentation

Quarkus OpenTelemetry auto-instruments:
- HTTP server (incoming MCP requests)
- REST clients (Prometheus, Loki)
- Kubernetes client HTTP calls (via OkHttp instrumentation)

### Custom Spans

For diagnostic tools, add manual spans to mark each diagnostic step:
- Add `@WithSpan` annotation on diagnostic service methods
- Or inject `Tracer` and create spans manually for finer control

**Example in `KafkaClusterDiagnosticService`:**
```java
@WithSpan("diagnose.cluster.status")
private ClusterStatus gatherClusterStatus(...) { ... }

@WithSpan("diagnose.cluster.pods")
private List<PodSummary> gatherPodHealth(...) { ... }

@WithSpan("diagnose.cluster.logs")
private LogResult gatherLogs(...) { ... }
```

### No New Files

This is primarily dependency + configuration + annotations on existing diagnostic services.

## Verification

1. `mvn clean test` — tests pass (tracing disabled by default)
2. Run with Jaeger/OTLP collector — invoke `diagnose_kafka_cluster`
3. Verify trace shows parent span with child spans for each diagnostic step
4. Verify K8s API calls appear as spans
