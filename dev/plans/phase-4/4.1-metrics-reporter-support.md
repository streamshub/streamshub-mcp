# 4.1 Strimzi Metrics Reporter Support

## Context

Strimzi Metrics Reporter is the modern replacement for JMX Exporter. It runs as a Kafka plugin (not a Java agent), exposes metrics directly on port 8080 via HTTP, and has lower overhead. The current MCP pod-scraping provider assumes JMX Exporter format (port 9404). We need to support both and auto-detect which is available.

## What Changes

### Modified Files

**`common/src/main/java/io/streamshub/mcp/common/metrics/PodScrapingMetricsProvider.java`** (or equivalent):
- Add auto-detection logic: try Metrics Reporter port (8080) first, fall back to JMX Exporter port (9404)
- Detection: inspect pod container ports — Metrics Reporter uses port name `metrics` on 8080, JMX Exporter uses `tcp-prometheus` on 9404
- Both expose Prometheus text format, so parsing is identical — only the port/path differs
- Cache detection result per pod to avoid repeated checks

### Configuration

**`strimzi-mcp/src/main/resources/application.properties`:**
```properties
# Metrics scraping: auto-detect (default), metrics-reporter, jmx-exporter
mcp.metrics.scraping.format=auto
```

### Key Notes

- Metrics Reporter exposes metrics at `/metrics` on port 8080 (same as Kafka HTTP endpoint)
- JMX Exporter exposes metrics at `/metrics` on port 9404
- Both use Prometheus exposition format — no parser changes needed
- The `metricsConfig` in Kafka CR tells which is configured: `type: jmxPrometheusExporter` vs `type: kafkaMetricsReporter`
- Reading the Kafka CR's `metricsConfig.type` is the most reliable detection method

### Test Files

- Update `PodScrapingMetricsProviderTest.java` — test auto-detection with both port configurations
- Test fallback when one format unavailable
