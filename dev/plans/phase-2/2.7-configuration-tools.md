# 2.7 Configuration Tools

## Context

Customers frequently need to understand the effective configuration of a Kafka cluster (CR spec + referenced ConfigMaps) and compare configurations between environments (prod vs staging). Currently there's no tool for this.

## New Tools

### `get_kafka_cluster_config`
- **Parameters:** `namespace` (optional), `clusterName` (required)
- **Returns:** Effective configuration assembled from:
  - Kafka CR spec: version, replicas, listeners, storage, resources, JVM options
  - Referenced ConfigMaps: `metricsConfig` (metrics configuration), logging ConfigMaps
  - Entity Operator config (if deployed): Topic Operator and User Operator settings
  - Cruise Control config (if deployed): goals, capacity, auto-rebalance settings
  - Kafka Exporter config (if deployed): topic/group regex filters
  - Node pool overrides: per-pool config, resources, storage
- **Service:** `KafkaConfigService.java`

### `compare_kafka_clusters`
- **Parameters:** `namespace1`, `clusterName1`, `namespace2` (optional, defaults to namespace1), `clusterName2`
- **Returns:** Side-by-side diff:
  - Fields present in both: show values, flag differences
  - Fields present in only one: flag as missing
  - Categories: version, listeners, storage, resources, replicas, Kafka config, JVM options
  - Summary: total differences count, categorized by severity (breaking vs cosmetic)
- **Service:** `KafkaConfigService.compareConfigs(cluster1, cluster2)`

## New Files

- `KafkaConfigService.java` — `@ApplicationScoped`, assembles effective config, computes diffs
- `ConfigurationTools.java` — `@Singleton`, `@Guarded` (or add to existing `KafkaTools.java` if small enough)
- `KafkaEffectiveConfig.java` — DTO for assembled config
- `KafkaConfigDiff.java` — DTO for comparison result
- `KafkaConfigServiceTest.java`, tests

## Key Design

- Config assembly: Read Kafka CR → for each ConfigMap reference, read the ConfigMap → merge into unified view
- Diff engine: Flatten both configs to key-value maps → compute symmetric difference
- Referenced ConfigMaps need `get` on `configmaps` — already in ClusterRole
- Handle missing clusters gracefully (one or both not found)
